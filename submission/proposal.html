Context

The Problem NHAI Asked Us to Address

NHAI operates 1.46 Lakh km of national highways. Each kilometre contains retroreflective assets — pavement markings, gantry signs, delineators, road studs — that must meet IRC 67 / IRC 35 minimum RL thresholds throughout their service life. When those thresholds fall, night-time visibility drops and accident risk rises.

Observed outcome: because each approach is either costly or slow, a significant share of the NH network is not measured on a regular cadence. A smartphone-based data-capture workflow could complement existing methods — not replace certified instruments — by broadening the sampling base at low marginal cost.

What the NHAI problem statement asks for

  1. Retroreflectivity measurement by vehicle / drone-mounted equipment, to improve speed of coverage.
  2. RL determination using AI & ML techniques on camera data.
  3. A combination of the above.

ReflectScan addresses all three directions while explicitly acknowledging:

  • This is a prototype designed to prove the data-capture and computation pipeline;
  • Accuracy of RL output depends on calibration, which is a scheduled next step;
  • The system is intended to augment IRC-compliant instruments, not substitute them in their certified role.

Data flow for a single capture

  1. Camera delivers a YUV420 frame while the LED flash is ON (illuminated).
  2. Next frame is captured with flash OFF (ambient baseline).
  3. YOLOv8n returns a bounding box around the road marking. (If model asset is absent, a brightest-patch heuristic is used.)
  4. Y-plane pixels inside the box are averaged for both frames.
  5. An OECF lookup table converts pixel values to cd/m². The prototype ships with a generic gamma-2.2 curve; per-device calibration is pending.
  6. Δ Luminance = illuminated − ambient.
  7. Flash illuminance at marking = flash_lumens / (4π × distance²). Uses a default flash lumen value; per-device photometric calibration is pending.
  8. RL = (ΔL ÷ illuminance) × 1000. Clamped to [0, 2000].
  9. GPS-tag, classify (with UNCAL fallback), persist to SQLite, queue for upload.
Accuracy caveat
Every step above is deterministic and reproducible from source. The numerical RL output, however, depends on two device-specific reference values — the per-phone OECF curve and the calibrated flash lumen output — that the current prototype uses generic defaults for. Establishing accuracy therefore requires a controlled calibration campaign, which is described in the "Validation Plan" section of this document.
Methodology

How ReflectScan Attempts to Estimate Retroreflectivity

Retroreflectivity (RL) is defined in ASTM E1710 as the coefficient of retroreflected luminance — luminance of the marking in cd/m² divided by the illuminance reaching the marking in lux, scaled to mcd/m²/lux. ReflectScan's intent is to compute an estimate of this quantity on commodity smartphones using the same optical relationship that a certified retroreflectometer uses.

The optical chain

  1. Illumination: phone LED delivers lumen output (nominal value; per-phone calibration required).
  2. Target: road marking glass beads retroreflect the light back toward the phone.
  3. Sensor: camera's Y-plane captures luminance-proportional pixel values.
  4. OECF curve: maps 0–255 pixel values back to photometric cd/m². Generic gamma curve today; per-device table needed for accuracy.
  5. Ambient subtraction: flash-OFF frame is subtracted to isolate the retroreflected component.

Core formula

ΔL   = L_illuminated − L_ambient
E    = flash_lumens / (4 π d²)
R_L  = (ΔL / E) × 1000

Units: L in cd/m², E in lux, R_L in mcd/m²/lux.

Clamp: output bounded to [0, 2000]. Real thermoplastic paint tops out near 500; the 2000 ceiling catches sensor saturation without producing runaway values.

Why we prefer a physics-derived approach over pure ML regression

Pure ML / CNN regressionReflectScan (physics + ML detector)
Learns a function photo → RL number; weights are opaque. Applies ASTM E1710 relationship directly; every step visible and reproducible.
A single unrepresentative training example can bias all outputs. Calibration curves are derived, once, from known physical targets under controlled conditions.
No natural mechanism to decline a reading; always outputs a number. Refuses to emit a classification when conditions are invalid (UNCAL tag).
Transfer to new paint formulations / new cameras typically requires retraining. New phone model → a repeatable calibration procedure; core algorithm unchanged.

What ML does in our design

The YOLOv8n TFLite model's role is detection, not RL estimation — it locates the marking in the frame. Separation of concerns: ML locates where to sample; physics (subject to calibration) computes the number from those samples. If the model asset is absent, a brightest-patch heuristic takes over — under flash conditions, retroreflective paint is typically much brighter than surrounding asphalt, so the brightest patch in the lower frame tends to coincide with the marking. Robustness of this heuristic across lighting conditions is itself subject to validation.

Design position: ReflectScan is architected so that the RL pipeline is auditable end-to-end. It does not claim current numerical parity with IRC-certified instruments. Calibrating the pipeline to achieve that parity is the primary work item in the proposed pilot.
Innovation 30 MARKS

What Distinguishes ReflectScan

Four design contributions

1. Physics-derived RL pipeline, not ML-regressed RL

Academic proposals frequently train a CNN to output RL directly from an image. This is a black box and resists audit. ReflectScan instead computes RL using the same physical relationship a certified retroreflectometer uses — luminance delta, OECF conversion, inverse-square-law illuminance. The result is a reproducible, explainable number whose correctness under calibration an optical engineer can examine.

2. Explicit uncertainty handling — the UNCAL tag

When flash differential is unavailable (strong daylight, torch hardware failure), the system refuses to output a SAFE / WARNING / CRITICAL classification and emits UNCAL instead. RL is clamped to the physically plausible ceiling [0, 2000] mcd/m²/lux; backend aggregation excludes UNCAL points from segment averages. Commercial retroreflectometers do not routinely expose their own uncertainty to the caller; ReflectScan does.

3. Zero-marginal-cost fleet deployment model

NHAI already operates thousands of patrol vehicles. The software runs on any Android 8+ phone with a flash. The projected marginal cost of deploying a new surveying unit is ~₹1 500 (a vibration-damped windshield mount). This is a design claim about deployability; actual field durability of the mount and smartphone workflow remains to be observed at pilot scale.

4. Offline-first data pipeline

Rural NH sections have intermittent coverage. Every capture writes to on-device SQLite first; uploads occur in batches when connectivity returns. This behaviour was exercised in the 22 April 2026 prototype drive and functions as designed in our code-level tests.

Prototype Testing & Validation Scope

Scope of this section
This section explicitly describes what has and has not been tested, so the remainder of the document can be read without ambiguity. The ReflectScan prototype's current testing is pipeline-level only.

What has been tested

  • Code-level tests. 27 backend pytest tests and 24 mobile Dart unit tests execute on every build; all passing at time of submission.
  • Offline data-capture pipeline. On 22 April 2026, the app was installed on a physical Android device, mounted inside a vehicle, and run on local road conditions in Ahmedabad for approximately 14 minutes. The exercise produced 82 GPS-tagged captures that were written to the on-device SQLite store, confirming the camera, flash control, detection fallback, classification, and offline-persistence paths execute end-to-end.
  • Error and edge-case handling. Torch failure breaker, GPS-delayed start, permission-denied flows, and missing-model fallback (brightest-patch heuristic) were all observed to behave as designed during the prototype drive.

What has not been tested

  • Measurement accuracy against an IRC-certified retroreflectometer. No comparison reading from a Delta-E LTL-XL, Zehntner RETRORED, or equivalent has been collected. Any RL number produced by the current prototype is therefore indicative, not authoritative.
  • Operation on NHAI-owned national highways. The 22 April drive was on local Ahmedabad roads, which contain comparable pavement markings but were not operated on under NHAI supervision, and do not constitute a national-highway pilot.
  • Per-device OECF calibration. The prototype uses a generic gamma-2.2 curve. A proper calibration requires controlled-illumination reference targets.
  • Photometric flash calibration. The flash lumen value is a nominal default for mid-range Android hardware; it has not been measured per device.
  • Wet, foggy, and low-light conditions. Design intent is documented; empirical behaviour across these conditions remains to be measured.
  • Mount durability at highway cruise speeds. Prototype drive was at urban speeds; high-speed vibration / wind-load effects on capture quality are not yet characterised.
  • Multi-device consistency. All captures were collected on one device model. Inter-device variance is unknown.

What the 82 captures do and do not demonstrate

The drive does demonstrateThe drive does not demonstrate
End-to-end software pipeline works on real hardware under real road conditions. That the RL numbers produced are within any stated tolerance of a certified instrument.
Flash strobe, GPS fix, frame pairing, detection fallback, and SQLite persistence function together. That the chosen SAFE/WARNING/CRITICAL thresholds classify this device's output correctly.
GPS accuracy sufficient for 100 m segment aggregation on local roads. Behaviour on expressways, in rain, in fog, or at speeds > 50 km/h.

Bottom line. ReflectScan at submission time is an end-to-end working prototype whose software pipeline has been exercised on local Indian road conditions. It is not an accuracy-validated retroreflectivity measurement instrument. Establishing accuracy against IRC-compliant standards is the subject of the proposed Phase 0 calibration and Phase 1 pilot.

Path to certified measurement

Validation Plan & Next Steps

This section describes how ReflectScan would progress from a working prototype to a tool whose outputs can be trusted by NHAI engineers.

Phase 0 — Controlled calibration (Month 1)

  1. Procure a reference instrument — a handheld retroreflectometer conforming to IRC 67 / IRC 35 / ASTM E1710 (e.g. Delta-E LTL-XL, Zehntner RETRORED). Equipment can typically be hired or borrowed from a state PWD / NHAI regional office / civil engineering department.
  2. Build an OECF lookup table per target device model. Photograph a standard reflectance chart (X-Rite ColorChecker or similar) under controlled illumination; regress pixel value against known luminance; store as a 256-entry table.
  3. Calibrate flash lumen output per device using a calibrated lux meter at a known distance.
  4. Establish reference targets. At least 10 physical markings spanning the RL range (freshly painted to badly faded), each measured with the reference instrument.
  5. Fit correction parameters. Align ReflectScan's output to the reference instrument via a simple linear or piece-wise regression, documented in the repository.

Phase 1 — NHAI pilot on live highway (Month 2–3)

  1. Deployment partner: one NHAI regional office, one patrol route (~200 km, mixed 4-lane / 6-lane), ~20 patrol phones.
  2. Ground-truth subset: a sub-sample of ReflectScan segments (e.g. 50 segments) re-measured with the handheld reference instrument within 48 hours of capture.
  3. Metrics. Report mean absolute error (MAE), bias, and 95% confidence interval between ReflectScan and reference RL per segment. Publish results transparently in the final pilot report.
  4. Failure-mode logging. Instances where ReflectScan emits UNCAL or its RL disagrees materially from the reference are flagged for post-mortem.
  5. Acceptance criteria for continuing. The pilot succeeds if ReflectScan identifies the same CRITICAL segments as the reference instrument (binary classification recall > agreed threshold, e.g. > 0.8), even if absolute RL has residual error.

Phase 2 — Integration and broader rollout (Month 4+)

  1. Integration with NHAI Road Asset Management System so that CRITICAL findings auto-open maintenance tickets.
  2. User training. Regional engineers learn the dashboard; patrol staff learn the 5-minute app workflow.
  3. Incremental rollout to additional states, each subject to a repeat of Phase 1-style validation on local reference markings.
  4. Benchmarking against IRC 67 / IRC 35 pass-fail criteria at each rollout step.

Road studs / delineators Add a road_stud class. Same pipeline, different target geometry.
Predictive maintenance Once enough time-series data accumulates, regression models on per-segment RL decay can anticipate threshold crossings.
